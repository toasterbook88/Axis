# AUTH-2: Node Identity Authority

## 1. How Nodes Are Identified

AXIS nodes carry **three identity signals**, layered from most stable to most transient:

| Signal | Source | Scope |
|--------|--------|-------|
| `stable_id` | OS-level hardware identity (Linux `/etc/machine-id`, Darwin `IOPlatformUUID`) or operator-override in `nodes.yaml` | Cross-session, cross-rename |
| `hostname` | Observed at runtime (`os.Hostname()` locally, `hostname` or `uname -n` remotely) | Per-boot, network-dependent |
| `name` | Operator-assigned logical label in `nodes.yaml` | Purely config-local |

### Source of Truth by Field

- **`NodeFacts.Name`** — always the config-assigned logical name (`config.NodeConfig.Name`). It is the primary key used in snapshot tables, placement decisions, and reservation ledgers.
- **`NodeFacts.Hostname`** — the *observed* hostname from the target machine. For remote nodes this is probed via SSH (`hostname` command, falling back to `uname -n`). For local nodes it is `os.Hostname()`.
- **`NodeFacts.Identity`** — an optional `*NodeIdentity` struct containing `StableID` and `Source`. Generated by OS-specific probes (see below).

### Config vs. Observed Identity

The config loader (`internal/config/config.go`) requires `name` and `hostname` but treats `stable_id` as optional:

```go
type NodeConfig struct {
    Name     string `json:"name" yaml:"name"`
    Hostname string `json:"hostname" yaml:"hostname"`
    StableID string `json:"stable_id,omitempty" yaml:"stable_id,omitempty"`
    // ...
}
```

Validation normalizes `StableID` via `models.NormalizeStableID()` (lowercase, trimmed) but does **not** enforce uniqueness across the node list.

## 2. Who Generates Node Identity

### Local Identity

`internal/models/locality.go` builds the local machine identity once per process via `sync.Once`:

1. `os.Hostname()` → `hostname` / `shortHostname`
2. `net.InterfaceAddrs()` → local interface IPs (loopback excluded)
3. `detectLocalStableID()` → reads `/etc/machine-id` (Linux) or runs `ioreg` for `IOPlatformUUID` (Darwin)

The stable ID is cached for the lifetime of the process (`localStableIDCached`).

### Remote Identity

`internal/facts/identity.go` probes remote nodes during SSH fact collection:

- **Linux:** `cat /etc/machine-id 2>/dev/null || cat /var/lib/dbus/machine-id 2>/dev/null`
- **Darwin:** `ioreg -rd1 -c IOPlatformExpertDevice ... | awk -F '"' '/IOPlatformUUID/ {print $4; exit}'`

If the probe fails, `NodeFacts.Identity` is `nil`. The remote collector does **not** fall back to hostname for identity.

### Discovery / Beacon Identity

`internal/discovery/udp.go` emits beacons containing:

```go
Beacon{
    Name:     localName,      // config name if local, else os.Hostname()
    Hostname: hostname,       // os.Hostname()
    StableID: localStableID,  // models.CurrentLocalStableID()
    // ...
}
```

Receivers deduplicate beacon nodes against the config map by `stable_id` first, then by `name` (`discoveredNodeKey`).

## 3. Where Identity Is Used for Placement Locality Matching

Placement locality is determined by `models.IsLocalNode(n NodeFacts) bool` (`internal/models/locality.go`). It returns `true` if **any** of the following match the local machine:

1. **Stable ID equality** — `n.Identity.StableID == local.stableID` (both normalized)
2. **Hostname match** — `n.Hostname` matches local hostname or short hostname
3. **Address match** — any non-link-local address in `n.Addresses` matches a local interface IP

`models.IsLocalConfig(name, hostname, stableID)` performs the same logic against a raw config entry, ignoring the logical `name`.

### Placement Consequences

- `internal/placement/selector.go`: `SelectBestNode` sets `PlacementDecision.IsLocal = models.IsLocalNode(best)`.
- `internal/placement/explain.go`: locality gates Apple Foundation Models (`requiresAppleFoundationModels` is local-only).
- `internal/placement/ranker.go`: local nodes receive a +10 `localBonus` in the fit score.
- `internal/placement/selector.go`: Ollama and Apple Foundation Model workloads append reasoning such as `"local node preferred for ollama"`.

## 4. Ambiguity Between Configured Name and Observed Identity

### Ambiguity 1: Logical Name ≠ Observed Hostname

The config `name` is purely a label. It is **ignored** by `IsLocalNode` and `IsLocalConfig`. This means:

- An operator can name a node `"mac-studio"` while the observed hostname is `"studio.local"`.
- Locality matching works as long as hostname or stable_id aligns.
- **Risk:** If the operator renames a node in `nodes.yaml` but keeps the same stable_id, locality still works, but reservation ledger entries keyed by the old `name` become orphaned.

### Ambiguity 2: Hostname Drift

- Remote hostnames are probed live. If a node changes hostname (e.g., DHCP lease change), the next fact collection captures the new hostname.
- `snapshot.Build` does **not** reconcile hostname changes against the config map; it trusts the observed `NodeFacts.Hostname`.
- **Risk:** If hostname drifts and stable_id is absent, locality matching may fail until the config is updated.

### Ambiguity 3: Duplicate Stable IDs

The config validator does not reject duplicate `stable_id` values. If two config nodes share the same stable_id:

- `discoveredNodeKey` collapses them into one entry during UDP discovery.
- `snapshot.Build` deduplicates by stable_id (config nodes win, then mesh nodes).
- **Risk:** The second config node is silently dropped from the snapshot, which can cause placement to ignore a valid node without warning.

### Ambiguity 4: Empty Stable ID on One Side

If the config supplies a `stable_id` but the remote node probe fails to return one (e.g., permission-denied on `/etc/machine-id`), `NodeFacts.Identity` is `nil`. Locality then falls back to hostname/IP matching, which may fail across NATs or VPNs even though the operator explicitly set a stable_id.

## Summary

| Layer | Authority | Mutable? | Used for Locality? |
|-------|-----------|----------|--------------------|
| `name` | Operator config | Yes (any time) | No |
| `hostname` | Observed OS state | Yes (per-boot/network) | Yes (fallback) |
| `stable_id` | OS hardware + operator override | No (hardware) / Yes (config) | Yes (primary) |

**Recommendation:** Always set `stable_id` in `nodes.yaml` for remote nodes that may change hostname or sit behind NATs, and verify that the remote probe can read the corresponding OS identity file.
